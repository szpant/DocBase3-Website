Q-Flow Customer Flow Management (7358)


What it isQ-Flow is Customer Flow Management tool that can help improve service delivery by managing queues electronically and then providing valuable decision making information.
SASM/Manager ProceduresManagement procedures can be located under SA(SM)/Manager - Accessing and Using Q-Flow
Key PrinciplesThe key principle that guides decision making on Classification code selection and Q-Flow usage is:
  • The Q-Flow system should be used to reflect the face to face interaction with customers. Any work completed outside of the customer’s visit (for example, back office work) will be updated in the Q-Flow system when the operator signs out and selects the relevant sign out reason. Each team member must login to QFlow at the commencement of their day and if not on counter sign out of QFlow using the appropriate sign out reason. If this reason changes throughout the day team members should login and update their sign out reason. At the end of each day all team member should use the end of day sign out reason.

QFlow classifications are used to provide more information about a customer interaction that is unable to be determined by source systems.
Access to Q-FlowAccess to QFlow is to be requested via email by SASM/Manager after the user has completed Q-Flow Training in Accelerate.

1.CSO's will be allocated a CSC for their Top Unit regardless of employment status (ie casual, permanent, contract). A region must not be allocated for the Top Unit unless approved otherwise.
2.Managers will be allocated their region as the Top Unit.
Q-Flow log-inQ-Flow can be accessed from an operator’s desktop by double clicking on the Q-Flow icon. Operators will select "Sign in automatically" when accessing Q-Flow and this will use their TMR network details to connect them to the Q-Flow system. This log-in process remains unchanged and is used by the operator at whichever CSC they are working from.

After login all operators should confirm their location prior to logging out or calling a customer.
Using Q-Flow on counterThere are business rules around how over the counter customer interactions are to be recorded using the Q-Flow system.

    Using Q-Flow as a Concierge
    Using Q-Flow as a Concierge
      When a customer presents at a CSC but isn't issued a ticket, the concierge should record the interaction in Q-Flow.

      There is a hidden button in the top left corner of the service console which can be pressed to record the interaction.



      Operators should then select one of the following interaction types:
      • Resolved enquiry - the concierge was able to resolve the customer's enquiry.
      • Service not provided - unable to provide the requested service, customer is not expected to return on the same day.
      • To return for service - customer needs to meet additional requirements (eg. complete a form) before a service can be provided. They are expected to return on the same day.

    Troubleshooting Guide
    Troubleshooting Guide
    Troubleshooting Guide
      Troubleshooting for common issues
      Troubleshooting for common issues
        Refer to the table below for a description of common issues and guidelines as to the required actions operators should take in the below circumstances:
        Wrong Classification Selected
        Wrong Classification Selected
          Description
          An operator has selected the wrong 'Classification' at the finalisation of a transaction.

          Action
          An incorrectly chosen Classification can be unselected by the operator prior to closing the ticket as required.

          The Q-Flow system forms only part of the information gathering system for the customers' purpose of visit.

          Multipurpose visit
          If a customer has more than one purpose of visit (for example renewing a driver licence and applying for an Industry licence) all classifications are to be entered at the end of the transaction.
        Unable to See a Frozen Ticket
        Unable to See a Frozen Ticket
          Description
          A customer returns to the counter and the ticket does not display in the 'Frozen' Queue.

          Action
          Frozen tickets will be displayed for those services currently available in the job function available to the operator. If a frozen ticket is not displayed after selecting the 'Frozen' filter, it may be that the operator has forgotten to ‘Freeze’ the ticket during the initial transaction, or it is for a service that is not in the operator's current job function. The function can be changed to 'LWT' to see all services.

          If the ticket is now visible and the operator has the appropriate skills to help the customer, they can ‘Unfreeze call’ and serve the customer. If the ticket is not visible, a new ticket can be created using ‘Direct Admit’ to finalise the transaction.
        Multiple Customers with 1 ticket
        Multiple Customers with 1 ticket
          Description
          More than 1 customer requiring service presents at the counter but only has 1 ticket.

          Action
          Operators should serve 1 customer on the ticket presented. Finalise all of the first customers transactions before closing the ticket and selecting 'Direct Admit' for the next customer from any service within the current Function.

          This is to be repeated for each individual customer.
        Single Customer with Multiple Tickets
        Single Customer with Multiple Tickets
          Description
          Where a single customer has taken multiple tickets to complete multiple transactions.

          Action
          Operators should complete all of the customer's transaction from the initial ticket called, classifying at the completion of the transaction.

          For the other ticket/s, operators should go to the 'Case List' Tab click on the ticket/s and click ‘Cancel case’.
        Dashboard is displaying a 4 hour wait time at the start of the day
        Dashboard is displaying a 4 hour wait time at the start of the day
          Description
          A ticket was not cleared from the 'Frozen' queue from the day before and the Real-Time Q-Flow Dashboard is now showing a 4 hour wait time at the start of the day.

          Action
          Tickets that remain in the 'Frozen' Queue at the end of day must be called and classified as 'Incomplete'. This ensures that the customer wait time and initial counter service time are recorded appropriately.

          Note:This wait time is removed from the reports available in the Report Access Portal (RAP) so do not have an impact on a location's ongoing statistics.
        End of Day Tickets in the Frozen Queue
        End of Day Tickets in the Frozen Queue
          Description
          In some instances 'Frozen' tickets will remain at end of day, these tickets indicate a customer that did not return to complete the transaction.

          Action
          Tickets that remain in the 'Frozen' Queue at the end of day must be called and classified as 'Incomplete'. This ensures that the customer wait time and initial counter service time are recorded appropriately.
        Unable to select a ticket from the Queue
        Unable to select a ticket from the Queue
          Description
          A customer has a ticket, is waiting to be served and a SASM asks an operator to call the ticket out of order. The operator is unable to select the customers ticket to call to their counter. This may indicate that the customer has selected to be served at either an accessibility counter or a counter that is only serving cash, and the operator trying to select the ticket is not at the appropriately designated counter.

          Action
          In this instance operators will need to advise the customer that they will serve them and 'Direct Admit' the customer.
        Blank page after successfully logging into Q-Flow
        Blank page after successfully logging into Q-Flow
          Description
          If after successfully logging into Q-Flow you are presented with a blank page, ie. menu bar does not display down the left hand side, this will be due to having other Internet Explorer windows open.

          Action
          Operators will need to close all Internet Explorer windows including TICA and STARS, Q-Flow can then be re-opened and should present properly.

          Operators are required to log into Q-Flow prior to opening any other Internet Explorer windows, particularly TICA.

    Requesting changes to Q-FlowRequests for change to Q-Flow functionality should be escalated through the CSC, QGAP or RSO manager to the Product Leadership team (PLT). The function request (e.g. change to kiosk buttons, new/altered job functions, or new classifications) with the reason for the change need to be provided.

    If PLT supports the requested change, the below process will be followed:
    • Request for change with PLT approval sent through to CSB Business Systems
    • CSB Business Systems will obtain a quote from the vendor for completing the changes
    • The quote is sent to PLT for approval
    • CSB Business Systems will liaise with the vendor to arrange a time for changes to be completed

    When considering the requested changes, PLT will need to determine who will fund the changes if funding is required.
    Technical Support and Supporting Information To report an issue with Q-Flow, use the 'Report an ICT Issue' function in Service Centre Online.

    Start the 'Brief summary' section with the word 'QFlow' and include a brief description of the issue, ensuring to include relevant key words like user, login, sign in, dashboard, kiosk, audio, printer, tv, monitor, hardware, etc.

    These keywords help with routing the issue to the correct area, to ensure timely action.

    For other Q-Flow enquiries, including requests for operator access refer: BS Q-Flow Support
